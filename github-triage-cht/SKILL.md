# GitHub Issue Triage

使用 label-based state machine triage 目前 repo 的 issues。從 git remote 推斷 repo。所有 GitHub 操作使用 gh。

## AI 聲明

triage 期間張貼到 GitHub 的每個 comment 或 issue body，都必須在最上方加入：

~~~markdown
> *This was generated by AI during triage.*
~~~

## 參考文件

- AGENT-BRIEF.md：如何寫 durable agent brief
- OUT-OF-SCOPE.md：.out-of-scope/ knowledge base 的運作方式

## Labels

| Label | Type | Description |
| --- | --- | --- |
| bug | Category | Something is broken |
| enhancement | Category | New feature or improvement |
| needs-triage | State | Maintainer needs to evaluate this issue |
| needs-info | State | Waiting on reporter for more information |
| ready-for-agent | State | Fully specified, ready for AFK agent |
| ready-for-human | State | Requires human implementation |
| wontfix | State | Will not be actioned |

每個 issue 應該剛好有一個 state label 和一個 category label。若 state labels 衝突，先指出衝突並詢問 maintainer 哪個正確，同時給出建議。

## State Machine

| Current State | Can transition to | Trigger | Action |
| --- | --- | --- | --- |
| unlabeled | needs-triage | Skill | 首次檢視，套用 label 前先給建議。 |
| unlabeled | ready-for-agent | Maintainer | issue 已明確且適合 agent，寫 agent brief comment 並套 label。 |
| unlabeled | ready-for-human | Maintainer | 需要人類實作，寫摘要 comment 並套 label。 |
| unlabeled | wontfix | Maintainer | spam、duplicate 或 out of scope，留言後關閉。 |
| needs-triage | needs-info | Maintainer | 規格不足，張貼 triage notes 與問題。 |
| needs-triage | ready-for-agent | Maintainer | 訪談完成且適合 agent，寫 agent brief 並套 label。 |
| needs-triage | ready-for-human | Maintainer | 訪談完成但需人類，寫摘要並套 label。 |
| needs-triage | wontfix | Maintainer | 決定不處理，留言後關閉；enhancement 可寫入 .out-of-scope/。 |
| needs-info | needs-triage | Skill | reporter 已回覆，提請 maintainer 重新評估。 |

issue 只能沿著這些 transition 移動。maintainer 可 override，但 skill 要指出不尋常之處。

## Invocation

maintainer 呼叫 /github-triage 後用自然語言描述需求，例如：Show me anything that needs my attention、Let's look at #42、Move #42 to ready-for-agent。

## Workflow: 顯示需要注意的項目

當 maintainer 要 overview，查詢 GitHub 並分成三組：

1. Unlabeled issues：沒有任何 labels，從未 triage。
2. needs-triage issues：需要 maintainer 評估或繼續評估。
3. needs-info 且有新活動：reporter 在最後一次 triage notes 後有留言。

每組顯示數量，組內由最舊到最新排序。每個 issue 顯示 number、title、age，以及 issue body 的一句摘要。讓 maintainer 選要深入哪一個。

## Workflow: Triage 特定 issue

### Step 1: 收集 context

在呈現任何建議前：

- 讀完整 issue：body、comments、labels、reporter、時間。
- 若有先前 triage notes，解析已確定內容。
- 探索 codebase，理解 domain、相關 interfaces、既有行為。
- 讀 .out-of-scope/*.md，檢查是否符合既有拒絕概念。

### Step 2: 提出建議

告訴 maintainer category 建議、state 建議、out-of-scope 相似記錄，以及 codebase 中相關發現。然後等待 maintainer 指示。

### Step 3: Bug reproduction

若分類為 bug，先嘗試重現：讀 reproduction steps、探索 code path、跑測試或 trace logic。成功就回報觀察行為與來源；失敗也要回報可能原因。資訊不足時，這是移到 needs-info 的強訊號。

### Step 4: Domain model session

若 issue 需要補規格，使用 $domain-model-cht 訪談 maintainer。

### Step 5: 套用結果

- ready-for-agent：張貼 agent brief comment。
- ready-for-human：張貼摘要，說明為何需要人類。
- needs-info：張貼 triage notes 與 reporter 問題。
- wontfix bug：禮貌說明並關閉。
- wontfix enhancement：寫入 .out-of-scope/、留言連結、關閉。
- needs-triage：套用 label；可視需要留言保存部分進度。

## Needs Info Output

~~~markdown
## Triage Notes

**What we've established so far:**

- point 1
- point 2

**What we still need from you (@reporter):**

- question 1
- question 2
~~~

問題要具體可行，不要只說請提供更多資訊。

## Resuming Previous Sessions

重新 triage 有先前 notes 的 issue 時，讀所有 comments、解析已確認內容、檢查 reporter 是否回答 outstanding questions、向 maintainer 呈現最新狀況，並從中斷處繼續 $domain-model-cht。
